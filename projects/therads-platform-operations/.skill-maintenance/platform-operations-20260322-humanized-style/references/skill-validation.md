# Skill 校验规则

本文件用于校验 `platform-operations` skill 的结构完整性、规则一致性和可维护性。

每次修改 `SKILL.md`、`references/`、`persona-registry.md`、审稿模板或 Threads 合规规则后，都要跑一次本校验。

## 一、自动校验

运行：

```bash
python3 scripts/validate_platform_operations.py /path/to/platform-operations
```

自动校验至少覆盖：
- `SKILL.md` 中引用的 `references/*.md` 是否真实存在
- `references/persona-expression-variance.md` 是否存在
- `persona-registry.md` 的 `active` 人设是否含占位符
- 人设状态值是否在允许范围内
- `CHANGELOG.md` 是否存在且包含日期化记录
- Threads 审稿与校验 reference 是否存在
- `references/usage-scope.md` 与 `agents/openai.yaml` 是否存在
- `references/content-output-enforcement.md` 是否存在
- 内容运营强化链路所需模板是否存在
- 内容队列、审稿包与人设栏目映射模板是否存在
- 周报模板、实验记录模板与 persona 启停 SOP 是否存在
- 月报模板、风险事件模板与多 persona 治理规则是否存在
- 阶段切换模板、风险应急 SOP 与资产归档规则是否存在
- 阶段目标追踪面板、恢复后观察清单与归档资产再启用规则是否存在
- 栏目生命周期评估、系列连续性总表与阶段退出清单是否存在
- 主线 / 副线配比规则、多周节奏压力测试模板与阶段迁移回顾模板是否存在
- 内容库存老化规则、多 persona 轮值排班模板与节奏失控回退清单是否存在
- 内容库存再生规则、多 persona 冲突仲裁模板与高风险栏目冷却期规则是否存在
- 高风险表达复发黑名单、栏目替代方案模板与账号异常后内容重启门槛是否存在
- 高风险表达替代表达库、栏目冷却期解封检查清单与内容重启后节奏升级规则是否存在
- 高风险表达近义变体禁用表、栏目解封后首轮排期模板与节奏升级失败回退阈值是否存在
- 近义变体触发后的自动回写规则、首轮解封栏目复盘模板与提频阶段分栏目升级顺序是否存在
- 自动回写完成度核对清单、首轮失败重新冷却 SOP 与提频阶段分 persona 升级顺序是否存在
- Threads 多账号运营增强链路文档是否存在
- Threads 多账号 Batch 01 的 `5` 份交付物文档是否存在
- Threads 内容运营强化第一阶段封板规则是否存在
- skill 是否已明确“默认维护核心、内容模板按需启用”
- 内容模板是否已挂上执行闸门，而不是只保留写法说明
- 去 AI / 人类纹理规则是否已挂入内容模板链，而不是只停留在抽象口号

## 二、人工校验

### 1. 结构完整性
- `SKILL.md` 的路由规则和资源导航是否一致
- 新规则是否放进合适的 reference，而不是继续堆在 `SKILL.md`

### 2. 实际使用对齐
- `SKILL.md` 是否已明确默认入口是维护核心，而不是默认进入 `P0-P5`
- `references/usage-scope.md` 是否把默认核心与可选内容模板包分开
- `references/usage-scope.md` 是否已把内容运营强化链路单独分层
- `agents/openai.yaml` 的描述与默认 prompt 是否与当前实际使用一致
- 若某模块长期只在 `SKILL.md` 中声明，却未进入真实工作链路，是否已降级为可选而不是继续默认扩张
- 内容输出时，是否存在真正的“生成前检查 + 生成后复核”闸门
- 账号基线卡、周排期与发布后复盘模板是否已接入内容运营强化链路
- 内容队列台账、统一审稿/发布包与人设栏目映射表是否已接入内容运营强化链路
- 周会/周报、实验记录与 persona 启停 SOP 是否已接入内容运营强化链路
- 月报/阶段复盘、风险事件记录与多 persona 共存治理是否已接入内容运营强化链路
- 阶段切换决议、风险应急恢复与内容资产归档是否已接入内容运营强化链路
- 阶段目标追踪、恢复后观察与归档资产再启用是否已接入内容运营强化链路
- 栏目生命周期评估、系列连续性总表与阶段退出清单是否已接入内容运营强化链路
- 主线 / 副线配比规则、多周节奏压力测试与阶段迁移回顾是否已接入内容运营强化链路
- 内容库存老化、多人设轮值排班与节奏失控回退是否已接入内容运营强化链路
- 内容库存再生、多人设冲突仲裁与高风险栏目冷却期是否已接入内容运营强化链路
- 高风险表达复发黑名单、栏目替代方案与账号异常后内容重启门槛是否已接入内容运营强化链路
- 高风险表达替代表达库、栏目冷却期解封检查与内容重启后节奏升级是否已接入内容运营强化链路
- 高风险表达近义变体禁用、栏目解封后首轮排期与节奏升级失败回退阈值是否已接入内容运营强化链路
- 近义变体触发后的自动回写、首轮解封栏目复盘与提频阶段分栏目升级顺序是否已接入内容运营强化链路
- 自动回写完成度核对、首轮失败重新冷却与提频阶段分 persona 升级顺序是否已接入内容运营强化链路
- 多账号运营增强链路是否仍保持“显式触发，不默认自动化”
- 多账号架构、项目层 skills 蓝图与 Batch 01 计划是否已接入入口与工作流
- 多账号矩阵总表、优先启动框架、共享钩子库框架、发布回写规格与 insights 规格是否已接入入口与工作流
- 若无真实运营痛点，是否已默认封板，不再继续预扩 Threads 模板主链
- `P0-P5` 与支撑模板是否显式要求先读 `content-output-enforcement.md`

### 3. Threads 证据完整性
- 平台、账号、文案规则是否优先引用官方来源
- 跨平台推断是否都标了 `Inference`
- 若更新了规则判断，是否同步更新“已核对日期”或来源说明

### 4. 人设治理完整性
- 自动匹配是否只面向 `active`
- 若当前刻意不保留 `active` 人设，规则里是否明确写了“不自动兜底”
- 有无把未补齐高风险字段的人设误设为 `active`
- 新字段是否和 `persona-governance.md` 保持一致
- persona-local 字段（例如 `output_limit`）是否只保留在对应 persona section，没有被抬升为全局默认规则
- 人类纹理字段（例如 `emotional_base`、`life_anchors`、`spoken_tics`、`texture_rules`）是否仍被视为 persona-local，而不是误写成所有人设通用

### 5. 去 AI 与人类纹理完整性
- `persona-expression-variance.md` 是否已被 `SKILL.md`、`content-output-enforcement.md` 或 `p0-persona-branding.md` 接入
- 是否明确区分“自然的人类纹理”和“故意写乱”
- 是否明确限制错字、空格代标点、错标点的使用边界
- 是否明确要求同模板跨 persona 使用时必须在情绪、生活锚点和句子节奏上出现差异

### 6. 审稿流程完整性
- 发布前审稿是否仍覆盖平台、账号、文案、人设、去重、连续性六层
- 输出模板是否仍保留结论、分级、依据和改写建议

### 7. 变更记录
- 是否更新 `CHANGELOG.md`
- 若属于高影响改动，是否写清影响范围

## 三、高影响改动

以下改动属于高影响，必须在变更记录中写明影响范围：
- Threads 平台规则判断口径变化
- 账号合规高风险项变化
- 文案风险分级变化
- 人设自动匹配门槛变化
- `active` / `draft` 状态定义变化
- 发布前审稿模板变化
- 默认使用边界变化：维护核心与可选内容模板包的划分变化

## 四、通过标准

### `pass`
- 自动校验无错误
- 人工校验无未解释风险

### `fail`
- 任一引用文件缺失
- 任一 `active` 人设含未解决占位符
- 当前无 `active` 人设，但规则未明确禁止自动兜底
- Threads 规则更新但无来源说明
- 高影响改动未记录
- `SKILL.md` 仍把 `P0-P5` 日更内容生产描述为默认入口
- `agents/openai.yaml` 与 `SKILL.md` 的实际使用定位不一致
- 内容模板存在，但未接入 `content-output-enforcement.md`
