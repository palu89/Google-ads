# 内容输出执行闸门

本文件用于解决一个实际问题：很多内容规则已经写进 skill，但在真实输出时并没有被执行。

原因通常不是规则不存在，而是它们只停留在说明层，没有变成“生成前必须做、生成后必须查”的动作链路。

## 一、适用范围

只要开始做以下任一事项，就必须先执行本闸门：
- 人设包装内容
- 方法论内容
- 每日早评
- 盘中跟踪
- 盘后复盘
- 晚间课程
- 单股深度分析
- 模板验证样稿

若当前任务是 skill 维护、persona 治理、Threads 审稿或结构校验，而不是内容产出，则不进入本闸门。

## 二、生成前强制检查

开始写正文前，至少完成这 8 项：

1. 任务归类
- 判断这次到底是 `内容运营对话`，还是 `skill 维护对话`
- 若不是内容产出，不要误入 `P0-P5`

2. 人设确认
- 读 `persona-governance.md`
- 读 `persona-registry.md`
- 明确当前使用的 `persona_id`
- 若人设不是 `active`，只能输出模板版、测试版或内部草稿

3. 模块确认
- 明确当前属于 `P0 / P1 / P2 / P3 / P4 / P5 / 单股分析`
- 不允许一篇内容同时混成多个模块

4. 目标确认
- 这篇要打中的用户是谁
- 这篇要解决的主痛点是什么
- 这篇要给用户带走什么具体收获

5. 连续性定位
- 这篇和上一篇是什么关系：延续、补充、修正、升级
- 这篇在当天内容链里扮演什么角色
- 若没有历史内容可接，至少声明“本次按单篇自洽处理”

6. 风险预判
- 是否包含实名、机构、履历、收益、翻身、财富自由、赌徒逆转等高风险叙事
- 若有，先读 `threads-copy-compliance.md`，必要时再读 `threads-account-compliance.md`

7. 表达纹理计划
- 读 `persona-expression-variance.md`
- 确认这篇的 `情绪底色`
- 确认这篇是否需要 `生活锚点`
- 确认这篇适合的 `句子粗糙度`
- 确认这篇是否需要 `口头纹理` 或 `轻量可见瑕疵`
- 若当前 persona 属于稳重理性簇，不要硬塞错别字、乱标点或空格错位

8. 结构公式计划
- 若这篇明确需要高张力标题、故事感推进、反差钩子或结构演练，读 `story-structure-formula-library.md`
- 先明确本次使用模式：`高表现版` 还是 `合规发布版`
- 至少确定：`标题公式`、`开头公式`、`是否使用转折公式`、`收尾公式`
- 若只是短讯提醒或极短帖，不强制使用完整公式链
- 若公式需要借强权威、信息差或结果验证制造张力，必须回到 `threads-copy-compliance.md` 先做风险改写
- 若用户没有明确要求只做结构拆解或高表现测试，默认使用 `合规发布版`

## 三、生成后强制复核

正文写完后，不允许直接交付，必须依次复核：

1. 表现力
- 读 `content-performance.md`
- 检查是否同时具备：`痛点`、`利他`、`互动`

2. 去重
- 读 `content-dedup-audit.md`
- 若没有历史内容可比，也要做内部自检去重

3. 连续性
- 读 `content-continuity.md`
- 若是单篇，也要检查是否自洽、是否说明与上下文的关系

4. 合规
- 至少读 `threads-copy-compliance.md`
- 涉及账号资料、实名、机构背景时，再读 `threads-account-compliance.md`

5. 人设一致性
- 回到 `persona-registry.md`
- 检查正文是否真的符合该 persona 的权威来源、语气、受众和禁区

6. 去 AI 与人类纹理复核
- 回到 `persona-expression-variance.md`
- 检查是否出现“太顺、太满、太像模板填空”的问题
- 检查生活锚点是否具体，而不是抽象地喊“真实生活”
- 检查错字、空格、口癖、错标点是否自然且节制
- 检查同一模板换 persona 后，是否真的在情绪、句长、停顿和互动方式上有差别

7. 结构公式复核
- 若这篇调用了 `story-structure-formula-library.md`，检查标题、开头、转折、收尾是否各自承担了不同功能，而不是都在重复“情绪张力”
- 检查结构张力是否来自痛点、误判、反转和提醒，而不是只靠夸张语气
- 检查是否把高风险人物背书、内幕感或收益验证包装成默认增长手法
- 检查本次模式有没有混乱：若标为 `合规发布版`，不能残留明显内幕感、收益炫耀或过猛戏剧化

## 四、执行失败的常见原因

以下任一情况，都会导致“规则明明写了，但输出时没执行”：

1. 只有参考文件，没有执行顺序
- 规则存在，但没有被挂进生成前/生成后的强制步骤

2. 模板只写结构，不写闸门
- `P1-P5` 模板告诉你怎么写，却没有要求先做人设、去重、连续性和合规检查

3. 没有最小执行记录
- 输出时没有留下“本次用了哪个 persona、做没做去重、做没做审稿”的痕迹

4. 没有失败分级
- 即使检查出问题，也没有 `BLOCK / REWRITE / REVISE / PASS` 这类处理级别

## 五、最小执行记录

每次内容输出，内部至少要完成以下记录；除非用户明确要求，不必全部展示给用户：
- 使用的人设
- 当前模块
- 主痛点
- 使用的纹理簇
- 是否使用生活锚点
- 是否使用结构公式
- 使用的是 `高表现版` 还是 `合规发布版`
- 若有，使用了哪组标题 / 开头 / 转折 / 收尾公式
- 是否做了去重
- 是否做了连续性检查
- 是否做了合规检查
- 最终处理级别：`BLOCK | REWRITE | REVISE | PASS`

## 六、强制处理规则

- 若未确认 persona，不得直接写正文
- 若未做去重，不得直接给“可发版”
- 若未做连续性检查，不得直接给系列内容
- 若未做合规检查，不得直接给 Threads 可发布结论
- 若复核结果为 `BLOCK` 或 `REWRITE`，不得伪装成已通过
