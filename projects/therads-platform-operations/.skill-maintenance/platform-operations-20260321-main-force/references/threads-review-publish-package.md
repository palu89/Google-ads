# Threads 审稿包与发布包模板

本文件用于统一两类交付：

- 审稿包：给出是否能发、为什么、要怎么改
- 发布包：给出最终成稿、发布说明和风险备注

这样可以避免每次审稿都临时拼字段，也避免“通过了审稿，但交付时缺关键信息”。

## 一、适用场景

- 单篇 Threads 成稿
- 批次中的某一篇进入审稿
- 一篇内容通过后准备交付发布

## 二、使用顺序

1. 先执行 `threads-review-workflow.md`
2. 若结论为 `REVISE` 或 `REWRITE`，先输出审稿包
3. 若结论为 `PASS`，再输出发布包
4. 同时更新 `threads-content-queue-ledger.md`

## 三、审稿包模板

```text
Threads 审稿包：
- 内容编号：
- 主题：
- persona：
- 模块：
- 审稿结论：PASS | REVISE | REWRITE | BLOCK
- 处理级别：
- 违规高风险：
- 推荐受限风险：
- 低风险优化项：
- 人设检查：
- 去重检查：
- 连续性检查：
- 依据来源：
- 建议改写：
- 下一步动作：
```

## 四、发布包模板

```text
Threads 发布包：
- 内容编号：
- 主题：
- persona：
- 模块：
- 当前状态：approved | scheduled | published
- 建议发布日期：
- 正文：
- 配套说明：
- 风险备注：
- 互动问题：
- 发布后关注点：
```

## 五、使用规则

- `BLOCK` 不生成发布包
- `REWRITE` 与 `REVISE` 先以审稿包为准，不伪装成“可发版”
- `PASS` 后仍要保留风险备注，不等于“毫无风险”
- 发布包里的正文必须和通过审稿的版本一致

## 六、常见错误

- 审稿结论写了，但没有给下一步动作
- 发布包只有正文，没有状态与风险备注
- `PASS` 后擅自改文案，却没有重新审稿
