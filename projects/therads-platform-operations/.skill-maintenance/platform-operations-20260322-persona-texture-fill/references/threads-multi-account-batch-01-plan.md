# Threads 多账号运营 Batch 01 计划

本文件用于定义 Threads 多账号运营的第一批落地动作。

`Batch 01` 的目标不是“立刻把 `10` 个号全部自动跑起来”，而是把多账号运营从想法变成可执行系统。

## 一、Batch 01 目标

完成以下 `5` 件事：

1. 定义 `10` 账户矩阵
2. 建立每个账户的最小骨架
3. 建立共享钩子治理
4. 建立发布与状态回写规格
5. 建立 insights 回收规格

## 二、交付物

### 1. 账户矩阵总表

至少包含：

- `account_id`
- `account_type`
- `primary_persona`
- `primary_track`
- `secondary_track`
- `target_frequency`
- `growth_goal`
- `status`

### 2. 账号基线卡骨架

对 `10` 个账户都建基线卡，但只要求：

- `2-3` 个重点账户进入优先运营状态
- 其他账户先保留骨架，不急着同时重启

### 3. 钩子库骨架

至少拆成：

- `观点钩子`
- `冲突钩子`
- `利益钩子`
- `悬念钩子`

并标注：

- 适用账号类型
- 适用 persona
- 风险等级

### 4. 发布规格

需要先定义：

- 发帖输入字段
- 发帖输出字段
- 失败回写字段
- 人工审稿前置条件

### 5. Insights 规格

至少明确回收：

- `views`
- `likes`
- `replies`
- `reposts`
- `quotes`
- `clicks`
- `followers_count`

## 三、执行顺序

### 第一步：矩阵结构

- 先决定 `10` 个账户各自角色
- 不允许 `10` 个号内容定位完全重复

### 第二步：优先级分层

- 选出 `2-3` 个优先运营账户
- 其余账号只保留骨架和发布预案

### 第三步：统一钩子治理

- 先建共用钩子库
- 再按账号类型做映射

### 第四步：发布和回写规格

- 先定义数据结构
- 再决定是否接 API 自动发帖

### 第五步：Insights 回收

- 先回收基础指标
- 再进入组合级比较和增长实验

## 四、成功标准

达到以下状态就算 `Batch 01` 完成：

- 已有正式 `10` 账户矩阵
- 已有优先运营账户名单
- 已有统一钩子库骨架
- 已有发布规格
- 已有 insights 规格
- 已明确下一批要做哪些项目层 skills

## 五、不属于 Batch 01 的事

以下内容不在 `Batch 01` 范围内：

- `10` 账户全自动并发
- 无审稿自动发布
- 全自动回复机器人
- 复杂 KPI 大盘
- 大规模 A/B 自动实验

## 六、风险点

- 账号定位重复，导致矩阵内耗
- 钩子高度重复，导致内容撞车
- 没有优先级，导致 `10` 个号一起发散
- 没有回写，导致增长只靠感觉

## 七、下一批衔接

`Batch 01` 完成后，下一批优先级是：

1. `threads-portfolio-orchestrator`
2. `threads-account-governor`
3. `threads-hook-engine`
4. `threads-publisher` 规格化
5. `threads-insights-collector` 规格化
